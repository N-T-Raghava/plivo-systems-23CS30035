Loss-Tolerant UDP Transport

System architecture and design summary for a deadline-constrained real-time media transport using overlapping XOR forward error correction.

System Architecture

The system transports a real-time stream in which the harness generates one frame every 20 ms. Each source frame contains a 4-byte big-endian sequence number and a 160-byte payload.

The sender does not simply forward the source packet. It transforms the stream into a custom transport protocol containing original data packets and two overlapping classes of XOR recovery packets. The hostile relay may drop, delay, reorder, or duplicate these packets before they reach the receiver.

The receiver interprets the custom packet header, stores the information it receives, reconstructs recoverable missing frames, and converts each successfully available frame back into the exact format expected by the harness player.

Wire Protocol

Every sender-to-receiver packet has a fixed size of 164 bytes: 

+----------------------+----------------------------------+
| 4-byte transport tag | 160-byte payload                 |
+----------------------+----------------------------------+

The 32-bit transport tag is stored in network byte order. The top two bits identify the packet type and the remaining 30 bits carry the source sequence number or parity-group base sequence.

Top Bits Packet Type Meaning
00 DATA Original source frame
01 PAIR XOR parity for two consecutive frames
10 TRIPLE XOR parity for three consecutive frames

The sender uses htonl() when constructing the header, and the receiver uses ntohl() when decoding it.

Sender Design

Every source frame is transmitted once as a systematic data packet. This means that when the original packet survives the hostile path, the receiver can immediately deliver it without decoding.

Pair Parity

For every two consecutive frames, the sender generates one XOR recovery packet: 

P2 = frame[a] XOR frame[a + 1]

If exactly one member of the pair is missing, the receiver can reconstruct it using the parity packet and the surviving frame.

Triple Parity

For every three consecutive frames, the sender generates a second recovery packet: 

P3 = frame[a] XOR frame[a + 1] XOR frame[a + 2]

The pair and triple boundaries overlap differently. This provides additional recovery equations for loss patterns that cannot be solved using pair parity alone.

Transmission Pattern

Over six source frames, the sender normally emits:

  • 6 original data packets;
  • 3 pair parity packets;
  • 2 triple parity packets.

The measured result is 1.88x bandwidth overhead, which remains below the required 2.0x limit.

Receiver and Recovery Architecture

The receiver maintains separate storage for source frames, pair parity packets, and triple parity packets. Packets are indexed by sequence number rather than arrival order, so network reordering does not change their interpretation.

Original Frame Path

When a DATA packet arrives, the receiver stores it if it is not already present and immediately sends the corresponding source frame to the player.

Pair Recovery

P = A XOR B

If A is missing:
A = P XOR B

If B is missing:
B = P XOR A

Pair recovery succeeds when the parity packet is available and exactly one source frame in the pair is missing.

Triple Recovery

P = A XOR B XOR C

If B is missing:
B = P XOR A XOR C

Triple recovery succeeds when the parity packet is available and exactly one source frame in the triple is missing.

Iterative Recovery

Recovery is not treated as a single isolated operation. A newly reconstructed frame may provide the missing information required to solve another parity equation.

Packet Arrives
→
Store Information
→
Attempt Recovery
→
New Frame Recovered
→
Attempt Further Recovery

The final receiver uses a small work queue to process these local recovery opportunities. This avoids repeatedly scanning all historical frames while still allowing recovery to propagate.

Reordering and Duplicate Handling

The hostile network may reorder and duplicate packets. The protocol therefore never assumes that arrival order matches source order.

Each packet carries explicit sequence information, and the receiver records whether each data or parity item has already been stored. Each source frame also records whether it has already been sent to the player.

As a result, reordered packets can still participate in recovery, while duplicate packets do not cause the same frame to be delivered repeatedly.

Why Proactive FEC Was Chosen

A reactive retransmission strategy requires the receiver to detect a loss, send a request through the hostile network, and then wait for the replacement packet to cross the hostile network again.

Both the request and the retransmission may themselves be delayed or dropped. Under a tight frame deadline, this round-trip recovery path can consume a significant portion of the available playout budget.

The final design instead sends recovery information proactively. No feedback traffic is required, and the measured runs therefore used zero feedback bytes.

Delay Strategy

The receiver does not deliberately hold an original or reconstructed frame once it becomes available. It sends the frame to the harness player immediately.

Therefore, DELAY_MS represents the total time available for network delay and FEC recovery rather than an additional receiver-side waiting period.

The lowest demonstrated valid delay in the sweep is 90 ms, but Profile B reaches exactly the 1.00% miss-rate limit at that point. An additional targeted run at 91 ms produced 13 misses, or 0.87%, on Profile B while retaining the same 1.88x bandwidth overhead. The recommended grading value is therefore 91 ms, which is only 1 ms above the lowest demonstrated valid delay while providing measurable margin below the validity boundary.

Experimental Design Evolution

Design Measured Overhead Outcome
Baseline 1.02x Losses were unrecoverable
Duplicate Transmission 2.05x Exceeded bandwidth limit
Pair XOR FEC 1.54x Valid, but residual loss patterns remained
Pair + Triple XOR FEC 1.88x Final design

The progression was driven by measured results. Full duplication improved reliability but exceeded the bandwidth cap. Pair parity reduced overhead and produced valid results, but Profile B still had 12 misses at both 100 ms and 140 ms.

Because increasing delay did not reduce those remaining misses, the final change used the remaining bandwidth budget for a second, differently grouped parity stream. At 100 ms, the final design reduced Profile A from 3 misses to 0 and Profile B from 12 misses to 7.

Failure Modes and Trade-offs

The scheme is designed to recover many isolated and overlapping packet losses, but it cannot recover arbitrary loss patterns.

  • If multiple source frames and the useful parity equations protecting them are lost together, insufficient information may remain for reconstruction.
  • Large correlated burst losses can exceed the recovery capability of the small pair and triple groups.
  • A packet that arrives after the frame deadline is useless even if its contents are correct.
  • Reducing the playout delay leaves less time for original and parity packets to arrive and makes timing failures more likely.

These limitations represent the central trade-off of the design: additional redundancy improves loss recovery, but the total measured traffic must remain below 2.0x.

The final system is a systematic, proactive, sequence-aware UDP transport. It spends the available bandwidth budget on overlapping forward error correction rather than retransmission, reconstructs frames locally at the receiver, and delivers each frame as soon as it becomes available.

The final measured configuration achieves 1.88x bandwidth overhead and demonstrates validity down to 90 ms across both supplied profiles. A targeted Profile B run at 91 ms produced a 0.87% deadline-miss rate, so 91 ms is recommended as the grading delay to retain measurable margin below the 1.00% deadline-miss limit while remaining close to the lowest demonstrated valid delay.
